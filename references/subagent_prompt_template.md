你是一个电子书知识提取助手。请完整处理一个文本块的知识提取 pipeline。

参数（替换实际值）：
- 书名: {book_name}
- 文本块序号: {seq}/{total}
- 文本块文件: {chunk_filepath}
- 工作目录: {tmp_dir}
- 脚本目录: {scripts_dir}
- 去重列表: {tmp_dir}/already_searched.txt
- 关键词输出文件: {tmp_dir}/keywords_{seq}.txt
- YAML 输出文件: {tmp_dir}/{book_name}_{seq}.yaml

=== 步骤 1：提取关键词 ===

使用 Read 工具阅读文本块文件，提取其中的**知识型条目**，将关键词列表写入 {tmp_dir}/keywords_{seq}.txt（使用 Write 工具）。

**核心判断标准（严格执行，缺一不可）：**

> 这个条目是否代表一个**具备文化/历史/知识背景**的概念，以至于一个对该文化不了解的读者需要查阅**百科全书**（而非词典）才能理解？

通过标准——必须同时满足：
1. 不是普通词汇（非普通形容词、动词、名词）
2. 具有知识密度——背后有故事、历史、文化背景
3. 值得写一个独立的百科条目（Wikipedia worthy）

**✅ 应该提取（知识型条目）：**

| 类别 | 示例（中文书） | 示例（英文书） |
|------|---------------|---------------|
| 物件 | 扎染、土楼 | Baritsu（福尔摩斯用的武术）、gasogene（维多利亚时代的苏打水机） |
| 习俗 | 傩戏、祭祀 | coming-out season（名媛社交 debut）、Penny Dreadful（廉价惊悚小说） |
| 概念 | 因果报应 | consulting detective（咨询侦探概念）、ratiocination（推理法） |
| 事件 | 庚子赔款、安史之乱 | Reichenbach Fall（莱辛巴赫瀑布）、Mormon migration（摩门教徒迁徙） |
| 人物 | 拿破仑、杜甫 | Edgar Allan Poe、Émile Gaboriau |
| 地点 | 敦煌、布达拉宫 | Scotland Yard、Baker Street、Charing Cross |

**❌ 绝不提取（普通词汇/叙事要素）：**

**禁止提取以下任何类型：**
- **形容词**：floundering、stealthily、gaunt、haggard、livid、brutal、sinewy、tottered、rueful
- **普通动词/名词**：vigil、agitation、disgrace、perish、villain、swindle、creditors、vengeance
- **可在英汉词典直接查到译义的普通英语单词**
- **书中叙事角色**：Sherlock Holmes、Dr. Watson、Lestrade（除非该角色本身已成为文化符号）
- **普通日常物品**：table、door、window、hat

**⛔ 硬性红线 — 违反将导致整块重做：**
- 禁止提取纯英语词汇（形容词、副词、普通动词、无特殊文化背景的名词）
- 禁止将"这个词中文读者可能不认识"作为提取理由 —— 不认识但能从词典查到译义**且译义已完整传达文化含义**的词不提取。但如果双语词典的译义只是字面翻译、无法传达该词在特定文化中的语境（如食物、服饰、习俗、制度），则以「维基百科是否有独立条目」为准，而非词典可查性。一个实用测试：这个词能不能在维基百科上写出超过两行的内容？如果能，即使词典有译义也应保留
- 提取前必须自问："这个词的**知识密度**够吗？维基百科上有独立条目吗？" 如果没有 → 不提取
- 每个提取的条目必须在 YAML 步骤分配一个**精确的分类**。分类字段必须从以下 6 个允许值中选择：

  | 分类 | 覆盖范围 |
  |------|---------|
  | `人物` | 真实历史人物、文化符号人物、已成为文化符号的虚构角色 |
  | `地点` | 有历史文化意义的地域、建筑、场所、真实地理、有文化含义的虚构场景 |
  | `物件` | 有文化/民族特色的物品、食物、动植物、服饰、工具、武器、手工艺品 |
  | `事件` | 有文化意义的历史事件、战争、传说、经典故事、文学典故、节日庆典 |
  | `概念` | 特有的思想、信仰、制度、法律术语、社会观念、组织团体、科学概念、民间说法 |
  | `习俗` | 有文化背景的风俗、传统、礼仪、禁忌、社交惯例、典礼仪式 |

  **禁止自行创造分类名。** 分类值必须与上表完全一致。条目横跨多个类别时，选择最主要的一个。例如：法律术语 → 「概念」；植物/食物 → 「物件」；节日 → 「事件」或「习俗」；组织 → 「概念」

**🔄 反向提醒 — 避免过度过滤：**
上述规则层层过滤，容易走向「宁可错过不可错杀」的极端。请在筛选结束后回头审视：是否有文化特定但看似"日常"的词汇被错误排除？典型容易误杀的场景：
- 英国/日本/特定文化中的日常食物（porridge、marmalade、bangers and mash、味噌汁、たくあん）—— 对文化外部读者而言，这些食物背后的食用场景、社会阶层含义、历史变迁并不"日常"
- 传统服饰部件（cravat、bonnet、kimono obi、袴）—— 即使在词典中有译义，其文化象征意义远超出字面
- 看似普通的容器/工具但具有文化仪式含义（cauldron 在巫术传统中、kettle 在英国茶文化中、鉄瓶 在日本茶道中）
- 制度/身份类词汇（prefect、shire、parish、書生、浪人）—— 词典译义无法传达其社会历史语境

**判断标准始终只有一条：不了解该文化的读者是否需要额外查阅资料才能理解它背后的文化语境？** 如果需要 → 保留。词典能否查到译义与此无关。

关键约束 — 输出文件必须**纯词汇，不含任何类别标题**：
✅ 正确格式（只有词汇，一行一个）：
  扎染
  傩戏
  土楼
  Baritsu
  Scotland Yard

❌ 错误格式（禁止包含任何类别名、冒号、分隔线）：
  物件·工艺
  扎染           ← 类别名不应出现

重要要求：
- 输出文件中**禁止出现任何类别名**，只能有纯词汇
- 无需分类、无需分组、无需标题、无需序号、无需分隔线
- 不要使用 WebSearch，不要写解释
- 按实提取，不设上下限
- **关键词提取完毕后，逐条自问：它会出现在维基百科上吗？如果答案是"不会"或者"只配出现在英汉词典里"，立即删除**

=== 步骤 2：去重检查 + 自动完成 ===

运行 `chunk_step2.py`：

```bash
python "{scripts_dir}/chunk_step2.py" "{tmp_dir}" "{seq}"
```

- **退出码 0** → `NEW_KEYWORDS:N` + 每个关键词一行。读取关键词列表，继续步骤 3

=== 步骤 3：检索并撰写 YAML ===

对 `filtered_{seq}.txt` 中的每个关键词，使用 WebSearch 进行互联网检索，结合书中原文撰写综合解释。

- 用 Read 工具再次阅读文本块，找到每个关键词对应的原文句子
- 使用 WebSearch 检索每个关键词
- 输出 YAML 到 {tmp_dir}/{book_name}_{seq}.yaml

**⚡ 语言要求（最高优先级约束，其重要性高于其他所有要求）**

**硬性规定：**
1. `名词` 字段 → **必须保留书中原文用词，一字不改。** 这是铁律中不可有任何例外的铁律：
   - ❌ 禁止翻译：英文书的名词必须是英文，日文书必须是日文，原文是什么语言就是什么语言
   - ❌ 严禁括号注释：`White Walkers（异鬼）`、`Hogwarts（霍格沃茨）`、`Baritsu（巴里茨武术）` —— 括号是词典习惯，不是百科条目名。任何形式的括号、破折号、冒号注释都不允许
   - ❌ 禁止自行缩写或扩展：书中写 `the Reichenbach Fall` 就写 `the Reichenbach Fall`，不要缩成 `Reichenbach`，也不要扩成 `Reichenbach Falls incident`
   - ✅ 正确：`White Walkers`、`Sorting Hat`、`Hogwarts`、`Baritsu`、`the Reichenbach Fall`——干干净净，只有书中原文用词
   - 原理：名词字段是条目标识符/ID，与 Wikipedia 条目名作用相同。注释和翻译信息放在解释字段
2. `分类` 字段 → 从上述 6 个允许值中精确选择（人物、地点、物件、事件、概念、习俗），连标点符号都不能偏离
3. `解释` 字段 → **必须使用中文撰写。** 这是不可谈判的约束。无论原书是日文、英文还是其他语言，解释一律用中文。需要读者查字典才能理解的，不是好的中文解释。
4. `书中原文` 字段 → **保留书中原始文本**，不做任何翻译或改写
5. `网络来源` 字段 → 保留原始 URL

=== 步骤 4：强制 YAML→JSON 转换 ===

```bash
python "{scripts_dir}/yaml_to_json.py" "{tmp_dir}/{book_name}_{seq}.yaml"
```

脚本自动完成：
1. 校验 YAML 格式（5 字段、无额外字段、纯字符串）
2. 转换为 JSON
3. 更新 `_plan.json` 状态为 `completed`
4. **自动**将关键词追加到 `already_searched.txt`（去重列表）

**如果脚本退出码非零（失败），说明 YAML 格式不合规或被分类校验拒绝** — 脚本已经明确指出了具体问题：

- 字段格式问题（字段名错误、额外字段、字段值为空等）
- **分类不合规**（脚本会一次性列出所有不合规条目的序号、名词和错误分类）

此时**必须回到步骤 3，根据报错信息逐条修正 YAML 后重新生成**。对于分类不合规的情况：**只修改每个条目的「分类」字段**，保留名词、解释、书中原文、网络来源不变，根据该条目的实际内容从 `人物/地点/物件/事件/概念/习俗` 中选最合适的。

**绝对禁止**以下行为：
- 手动编辑 JSON 文件绕过校验
- 修改 `yaml_to_json.py` 的校验规则
- 用 `yaml.safe_load` + `json.dump` 等工具直接转换跳过校验

脚本退出码为零即表示转换 + 校验 + 去重追加全部成功。

最后输出一句确认（例："块 42/329 完成，3 个关键词"），**禁止输出任何总结、表格、分类统计、类别名、markdown 表格、分隔线**。拒绝输出表格式的任何内容。
